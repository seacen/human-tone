# 外部 skill 调研 · 值得吸收的设计机制（digest）

> 来源:深读 9 个开源"去 AI 味"项目提炼(blader/humanizer、op7418/Humanizer-zh、hardikpandya/stop-slop、MrGeDiao/shuorenhua、B1lli/remove-ai-flavor-writing-skill、LifelongLazyLearner/qu-ai-wei、OUBIGFA、worldwonderer/oh-story、hylarucoder/ai-flavor-remover)。
> **用途:这是我们设计的输入池之一,和 data-analysis 的架构理念平级——data-analysis 只是"通用+具体"的一个样本,下面这些同样要参与设计选项与决策,不能丢。**

## 一、值得偷的机制(每条标出处 + 它兑现哪条要求)

| 机制 | 出处 | 兑现 | 要点 |
|---|---|---|---|
| **逃逸链条追捕 / 变体封堵** | OUBIGFA | ③ 从错误引申规律 | 为同一母模式布多个子条目,逐条注明"禁止换成 X/Y 做变体逃逸"——堵住"禁'赋能'就换'助力'"的换皮。这是"从词到母模式"的落地形态。 |
| **宾语消歧 + 误杀白名单** | shuorenhua / qu-ai-wei | ② 高精度不误伤 | 同词按宾语定去留:`接住流量`留、`接住你的脆弱`删。配多义词清单 + boundary-cases"不改"样例。不误伤的核心。 |
| **LLM 改写 + 脚本确定性复扫** | oh-story / B1lli | ② 高召回 | 语义交 LLM,召回交正则脚本兜底到 0(扫"不是…而是"、破折号密度、句长方差、"很X"密度)。把召回从"靠 LLM 记性"变"机器兜底"。 |
| **看密度不看单次** | shuorenhua / qu-ai-wei | ② 不误伤 | 单次排比/成语/破折号不算 AI 腔,只惩罚"短段内机械堆叠、脱离内容、删掉不丢意"。给量化阈值。 |
| **规则按语体分档(九语体激活矩阵)+ 烈度倒转** | qu-ai-wei / oh-story | ② 不误伤 | 公文"依法予以处理"、学术"进行分析"是规范表达。用语体矩阵让每种文体只触发对应规则子集。同一情绪词长篇删、短篇留。 |
| **平庸化回滚** | OUBIGFA | 质量兜底 | "删完套话后文本太平太匀太像被清理过"本身是失败态——列可诊断症状(段落变薄、句长变均、视角被抹平),命中即回滚。防越去越干瘪。 |
| **强制自查回路** | blader / shuorenhua | ② 高召回 | 改写后模型必须列≥2 条残留 AI 腔再改一版。用模型自己的检测力打自己的草稿。纯 prompt 里最有效的召回补丁。 |
| **信息保真双向打钩** | qu-ai-wei | 保真 | 原文每条事实终稿要打钩,终稿每条事实原文也要打钩——防漏也防增(堵"注入人味时编造细节")。 |
| **What-NOT-to-flag / 保留人味清单** | blader | ② 不误伤 | 显式列"这些不是 AI 腔、别动"——业内最强的护栏之一。 |
| **PostToolUse hook 兜底** | oh-story | 高召回 | 确定性脚本挂 hook,写完自动复扫,`_after` 必须 0-blocker。 |

## 二、结构 / 工程理念

- **渐进式披露**(stop-slop):~2.6KB 主文件常驻 + 参考文件按需加载。对抗 qu-ai-wei(900+行)/OUBIGFA(640 行)的体量陷阱——"作者自己都担心模型执行不全"。
- **三级严重度分档**(shuorenhua):词表/规则按 must-fix / 建议 / 视情况分级。
- **可调力度轴** minimal/standard/aggressive(shuorenhua):把精度↔召回权衡交给场景,不写死。
- **真人 vs AI 门检**(qu-ai-wei):先判输入是不是 AI 写的再动手;作者自认这步最常被跳过——把 AI 稿误判成真人整篇不改代价最大。
- **eval:SF/SNF 双集 + 双模型交叉评判**(shuorenhua / oh-story):SF=该改的,量召回;SNF=一个字都不该动的,量精度(被改即误伤扣分)。改结果由另一模型盲评。回归门禁 `_after` 0-blocker。

## 三、别人普遍栽的坑(我们要主动避开)

1. **教学味太重**(humanizer/Humanizer-zh/hylarucoder):花大篇幅教"好中文怎么写",拉长指令、违背去除定位。→ 我们只保留"什么时候别改"的护栏。
2. **只有黑名单没有规律**(Humanizer-zh/hylarucoder):禁词表一禁就换皮,只有 OUBIGFA 真做了变体封堵。→ 我们必须母模式为主。
3. **召回全靠黑名单硬碰硬**:无强制回路、无机器兜底,长文必漏。→ 脚本 + 自查双路径。
4. **易误伤/过度改写**(stop-slop 杀光副词/禁所有破折号;hylarucoder 鼓励堆比喻反造新 AI 腔)。→ 带语境条件地禁。
5. **几乎都没 eval**:改写质量不可验证、改规则就回归。→ 我们自带 SF/SNF 双集。
6. **脚本本身有固有短板**:正则误命中("从北京到上海"命中"从…到…")、易被改写绕过、只能报告不能语义改写。→ 误报预算 + 脚本只报告不改写 + 冲突以模型为准。

## 四、data-analysis 样本的定位

data-analysis skill 贡献的是**"通用+具体如何分层"的架构理念**(薄入口 + 渐进披露 + 中立 resolver 自动枚举包 + SSOT 槽协议 + 独立 builder + 通用层零领域知识),不是内容。它和上面这些去 AI 味机制**互补、不互斥**:前者给"怎么组织",后者给"装什么、怎么保精度与召回"。设计时两边都要吸收。
